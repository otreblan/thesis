\section{Overview}

Fdtd Lucuma is a simulator made in C++26.
It can be used from the command line and from a gui made with Dear ImGui\cite{imgui}.
Internally it is structured using Entity Component System (ECS) with EnTT\cite{entt} and with swappable services using dependency injection.
From which the \texttt{frontend} (cli or gui) consumes one of the \texttt{backends}.
Currently it has 9 backends:

The sequential backend uses plain C++, while the parallel one uses Taskflow\cite{Huang_2022}.
For the GPU backend, Fdtd Lucuma implements the FDTD method using Vulkan\cite{vulkanspec} Compute Shaders written in Shader Slang\cite{10.1145/3197517.3201380}.

\section{Structure}

\begin{figure}[H]
	\centering
	\includesvg[width=1\textwidth]{images/flow.svg}
	\caption{Fdtd Lucuma's services graph}
	\label{fig:fdtd_lucuma_service_graph}
\end{figure}

Fdtd Lucuma is structured in services.
Each one is lazily instantiated the first time one of it's dependants needs to inject it as a dependency.
The services communicate with each other using signals via EnTT's dispatcher while also sharing data with EnTT's registry.
The registry is used to store the simulation data as components.

\section{Flow}
\begin{enumerate}
	\item The program starts.
	\item Basic services are instantiated.
	\item The backend to be instantiated is selected from the command line arguments. (As shown in Figure \ref{fig:fdtd_lucuma_service_graph})
	\item A new ECS entity is created and the simulation state is attached as a component.
	\item The electric sources are attached to new entities.
	\item The simulation iterates for a determined number of timesteps.
	\item The electric sources are deleted.
	\item The simulation state is deleted.
	\item The services are deleted in reverse order of creation.
\end{enumerate}

\section{CPU backends}
The Sequential backend is the simplest of the three.
At each step it updates each $\{x,y,z\}$ component of the magnetic field, then the components of the electric field, then it applies each source that exists in the EnTT registry and finally the boundary conditions of each top and botom of each component.

In comparison, the parallel backend uses Taskflow to create a task graph.
It allows for each component of the field updates to be parallelized as they do not have dependencies on each others.


\section{GPU backend (Vulkan)}
The Vulkan backend accelerates the simulation by delegating the field updates to a GPU.
